## Function-by-function changes

### `src/proxy.ts`

- **Imports preserved verbatim:** `import { type NextRequest } from 'next/server'`, `import { updateSession } from '@/utils/supabase/middleware'`.
- **`proxy(request)` function preserved verbatim:** still returns `updateSession(request)`.
- **`config.matcher` replaced** from the broad regex `'/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)'` to the narrow list `['/admin/:path*', '/host/:path*', '/login']`.

### Why this is safe with `updateSession()`

I verified `src/utils/supabase/middleware.ts` (read-only — not in my ownership) before making the change:

- It only reads/writes Supabase auth cookies — no server-side enforcement that depended on running on `_next/static`, `/`, `/display/*`, or `/player/*`.
- The redirect logic inside `updateSession()` only fires for paths starting with `/admin` or `/host`, or the literal `/login` — exactly the routes the new matcher covers.
- Public routes (`/`, `/display/*`, `/player/*`) now bypass middleware entirely — a deliberate latency win for the TV display and follower phones.

### `src/app/login/page.tsx`

- Removed `signup` from the `./actions` import. Only `login` is imported.
- Removed `const [mode, setMode] = useState<'login' | 'signup'>('login')`.
- `handleSubmit` now always invokes `login(formData)` directly (no `mode === 'login' ? login : signup` branch).
- Hardcoded the card title to `'Welcome Back'` and the submit button to `'Sign In'`.
- Removed the toggle button (`<button onClick={() => setMode(...)}>...</button>`) entirely.
- All other behaviour preserved verbatim: `Suspense`, search-params `next` handling, missing-Supabase-config error gate, error toast box, the email + password fields, `isPending` button loading state, the `router.push(redirectTo)` on success.
- No password-reset / forgot link existed in the original — none added.
